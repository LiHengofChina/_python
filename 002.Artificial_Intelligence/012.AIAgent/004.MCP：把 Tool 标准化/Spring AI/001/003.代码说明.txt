//========================================== MCP 算基础设施层吗？

可以这么理解。 在分层上大致是：

应用层    Agent 编排、对话、审批、RAG 流程
          ↓
基础设施  MCP Server（Tool 能力：SSH、监控、邮件…）
          ↓
外部系统  Linux、MySQL、邮件网关…

Tool 拆到 MCP 微服务后，职责清晰、独立部署、多客户端复用，确实更好管。和你做 Spring 微服务拆模块的思路一致。

//========================================== 客户端调 MCP 是普通 HTTP 吗？
//========================================== 客户端调 MCP 是普通 HTTP 吗？

底层走 HTTP，但不是随便调个 REST 接口。

SSE + HTTP（你 day_06 Demo）	//连 /sse，用 MCP 规定的 JSON 消息格式通信

STDIO						//本地子进程，不走 HTTP（Cursor 本地 MCP 常用）

Streamable HTTP				//新版 Spring AI 也支持

//========================================== 所以

传输层是 HTTP，协议层是 MCP（发现 Tool、调 Tool、返回结果都有固定格式），
不是你自己写的 GET /api/xxx 那种普通业务 API。

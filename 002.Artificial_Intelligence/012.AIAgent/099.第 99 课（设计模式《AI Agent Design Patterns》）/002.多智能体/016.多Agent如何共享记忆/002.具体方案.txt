
//========================================== 方案一： 黑板模式 + 状态路由
//========================================== 方案一： 黑板模式 + 状态路由

（1）Agent A 完成任务：不分享思考过程，只写当前结论
（2）黑板： Redis Hash：极短JSON状态，持久化中转站
（3）Agent B 按需读取：只拿最相关状态，精准一段JSON
（4）结果精准注入：当作工具返回值，不污染主上下文

核心原则：把 "别的Agent 的产出" 当作 "当前 Agent 的工具返回结果——精准路由"，
		上下文不越界。
/***
在真实项目里，所有Agent共享一个轻量级的中间状态存储
比如：Redis里面的一个Hash
当Agent完成一个任务，它不分享冗长的思考过程，
只把当前结论和下一步状态写到黑板上，
Agent B 去合不拢嘴这个状态时，拿到的只是一段极短的 JSON，
这叫上下文按需路由，把别Agent的产出当成当前Agent的工具返回结果，
*/

//========================================== 方案二： 长期共享记忆的克制使用
//========================================== 方案二： 长期共享记忆的克制使用

/**
错误的方法，是把记忆全部塞进 "向量数据库"
听起来很美，但是召回的 "几十个文本块" 拼进提示词。
token瞬间爆炸，模型还容易产生幻觉。
*/
正确做法：
	压缩代理：Agent的经验要进入共享区，必须先被大模型压榨成100字以内的纯结论性事实。

（1）Agent经验产出
（2）压缩代理（大模型压缩至100字以内）
（3）共享记忆区（向量）
（4）检索：元数据过滤 ——> 精准1条

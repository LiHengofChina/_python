//======================================================= RAG 是什么？
//======================================================= RAG 是什么？

RAG = Retrieval-Augmented Generation（检索增强生成）

/**
先 "查资料" ，再让 "大模型" 回答。

不让模型只靠 "脑子里记的东西" 瞎编，

而是先从 "你提供的文档/数据库" 里 "检索相关内容"，再基于 "这些内容" 生成答案。

*/

//======================================================= 为什么需要 RAG？
//======================================================= 为什么需要 RAG？

大模型有两个常见问题：

（1）不知道你的私有数据
	//没学过你们公司的运维手册、内部规范

（2）会胡编（幻觉）	
	//不知道也乱答，看起来很自信

//======================================================= RAG 的解决办法：
//======================================================= RAG 的解决办法：

用户提问
    ↓
先从"你的文档库"里搜相关内容
    ↓
把 "搜到的内容 + 用户问题" 一起给大模型
    ↓
模型根据 "真实资料" 回答

//======================================================= 与之前知识点的关系？
//======================================================= 与之前知识点的关系？

之前学过的 Embedding、词向量
        ↓
    文档也变成向量存起来
        ↓
    用户问题也变成向量
        ↓
    找最相似的文档片段（检索）
        ↓
    拼进 Prompt 给 GPT 生成答案

//======================================================= 所以
//======================================================= 所以

所以 RAG  "不是新架构" ，是 Embedding + 检索 + LLM 拼在一起。



//======================================================= 总结
//======================================================= 总结

（1）RAG = 先检索，再生成
	 本课你要记住的 3 句话

（2）解决模型不知道私有数据、容易胡编的问题

（3）流程：文档切块 → Embedding → 向量检索 → 拼 Prompt → LLM 回答

工单提交后，系统需要通知相关的审核用户，
提醒他们进行审批。这里涉及到两个主要功能：通知 和 审批操作。


//===========================================  通知机制
//===========================================  通知机制

（1）在用户提交工单后，系统会根据审批流程发送通知给相关角色的用户。
	这些用户可以是当前已经登录的用户，
	也可以是没有登录的用户（通知可以通过邮件、站内信等形式发送给未登录用户）。

（2）通知设计
	发送通知：当工单提交并进入审核流程时，
	系统应该根据 resource_group_approval_flow 表中配置的角色顺序，
	查找该角色对应的用户，并发送通知给他们。
	通知内容：通知中可以包含工单的基本信息，
	如工单ID、任务名称、提交人、SQL 内容等。
	已读/未读标志：通知需要有 已读 和 未读 状态，用户点击通知后将其标记为已读。

（3）通知列表和提醒（红点）
	红点提醒：在用户的右上角显示未读通知的红点，提示用户有新通知。
	通知列表：用户点击红点后，弹出通知列表，列出所有未读通知。每个通知包含工单信息和审批操作入口（“通过”或“拒绝”）。
	已读标记：用户点击通知后，将该通知标记为已读，同时进入审批详情页面。

（4）
	工单审批操作
	当用户点击某个通知时，系统应跳转到审批详情页面，展示该工单的详细信息，用户可以选择：
	通过：通过审核该工单。
	拒绝：拒绝审核该工单。

CREATE TABLE notification (
    id INT PRIMARY KEY AUTO_INCREMENT,
    user_id INT,  -- 收到通知的用户ID
    work_order_id INT,  -- 对应的工单ID
    message TEXT,  -- 通知内容
    is_read BOOLEAN DEFAULT FALSE,  -- 是否已读
    created_at TIMESTAMP,  -- 通知时间
    updated_at TIMESTAMP,  -- 更新通知时间
    FOREIGN KEY (user_id) REFERENCES users(id),
    FOREIGN KEY (work_order_id) REFERENCES work_order(id)
);

（5）
用户提交工单
    ↓
系统查询资源组审批流程
    ↓
系统根据角色顺序发送通知给相应用户
    ↓
用户点击通知
    ↓
进入审批页面（查看工单详情）
    ↓
用户选择“通过”或“拒绝”
    ↓
系统更新工单状态
    ↓
通知发起者并结束流程
